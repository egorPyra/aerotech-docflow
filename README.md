# Aerotech Docflow

Backend-сервис для запуска, контроля и архивирования сканирования документов.

Проект предназначен для сценария, в котором внешняя система (в дальнейшем — Planfix) создаёт задание на сканирование, backend ожидает новый PDF в папке сканера, проверяет файл, формирует итоговое имя, вычисляет SHA-256, переносит документ в архив и сохраняет состояние задания в SQLite.

На текущем этапе интеллектуальный анализ документов, OCR и AI не реализованы и отложены на отдельный этап.

---

## 1. Текущий сценарий работы

```text
Клиент / Swagger / будущая кнопка Planfix
        ↓
POST /scan
        ↓
Создание задания в SQLite
        ↓
Постановка задания в очередь
        ↓
Ожидание нового PDF в папке сканера
        ↓
Проверка готовности и PDF-сигнатуры
        ↓
Формирование нового имени
        ↓
Расчёт SHA-256
        ↓
Перемещение в архив
        ↓
Сохранение результата
        ↓
status = done
```

Пока настоящий драйвер Epson не подключён, сканирование имитируется помещением PDF в папку `data/incoming` после создания задания.

## 2. Что уже реализовано

### HTTP API

- FastAPI-приложение.
- Swagger UI.
- OpenAPI-схема.
- Health-check.
- Создание задания.
- Просмотр задания.
- История заданий.
- Отмена активного задания.
- Повторный запуск неудачного или отменённого задания.
- Пагинация истории.
- Стандартные HTTP-коды ошибок.

### Жизненный цикл задания

Реализованы статусы:

```text
accepted
waiting_for_file
file_received
archiving
done
failed
cancelled
```

### Работа со сканером через папку

- Ожидание нового PDF.
- Игнорирование PDF, существовавших до запуска задания.
- Проверка стабильности размера файла.
- Проверка, что файл не пустой.
- Проверка заголовка `%PDF-`.
- Таймаут ожидания.
- Очередь для одного физического сканера.

### Архивирование

- Формирование стандартизированного имени.
- Очистка запрещённых символов Windows.
- Обработка зарезервированных имён Windows.
- Защита от перезаписи.
- Добавление суффиксов `_01`, `_02` и далее.
- Расчёт SHA-256.
- Безопасное перемещение между дисками.
- Копирование через временный файл `.part`.
- Архивная структура по типу документа, году и месяцу.

### Постоянное хранилище

- SQLite.
- WAL-режим.
- Сохранение заданий после перезапуска.
- Простые встроенные миграции.
- Индексы по статусу, дате, `task_id` и внешнему ID.
- Пометка прерванных заданий как `failed`.

### Идемпотентность

- `external_request_id` уникален.
- Повторный идентичный запрос возвращает прежнее задание.
- Повторный запрос не запускает сканирование второй раз.
- Тот же `external_request_id` с другими данными возвращает `409 Conflict`.

### Отмена

- Можно отменить `accepted`.
- Можно отменить `waiting_for_file`.
- Нельзя отменить `file_received`, `archiving`, `done`, `failed`.
- Фоновая `asyncio.Task` отменяется по `request_id`.
- Статус сохраняется как `cancelled`.

### Повторный запуск

- Retry разрешён для `failed`.
- Retry разрешён для `cancelled`.
- Сохраняются `request_id` и `external_request_id`.
- Увеличивается `attempt_count`.
- Очищаются старые пути, SHA-256 и ошибка.
- Запускается новая попытка ожидания PDF.

### Автоматические тесты

Реализованы тесты:

- API;
- валидации;
- идемпотентности;
- SQLite;
- сохранения после переподключения;
- статусов;
- архива;
- SHA-256;
- ожидания нового PDF;
- таймаутов;
- отмены;
- retry.

Тесты используют временные папки и временные SQLite-базы и не затрагивают рабочие документы.

### `app/repository.py`

Работа с SQLite:

- создание базы;
- создание таблицы;
- миграции;
- CRUD заданий;
- идемпотентность;
- поиск по `request_id`;
- поиск по `external_request_id`;
- изменение статусов;
- подготовка retry;
- сохранение результата архивирования.

### `app/service.py`

Оркестратор обработки:

- запуск фоновой задачи;
- очередь одного сканера;
- ожидание файла;
- обновление статусов;
- архивирование;
- обработка исключений;
- отмена конкретной задачи.

- проверяет стабильность;
- проверяет сигнатуру;
- завершает ожидание по таймауту.

Модуль пока не отправляет команду аппаратному сканеру.

### `app/storage.py`

Файловые операции:

- очистка частей имени;
- формирование имени;
- контрольный код;
- SHA-256;
- создание папок архива;
- выбор свободного имени;
- безопасное перемещение PDF.

### `app/config.py`

Настройки:

- входящая папка;
- архив;
- SQLite;
- таймаут;

## 8. API

### `GET /`

Информация о сервисе.

### `GET /health`

Проверка, что приложение работает.

### `POST /scan`

Создание задания.

Пример:

```json
{
  "external_request_id": "manual-test-001",
  "task_id": 52418,
  "document_type": "UPD",
  "document_number": "001",
  "user_code": "IE",
  "context": {
    "source": "manual"
  }
}
```

Первый запрос возвращает `202 Accepted`.

Повторный идентичный запрос возвращает прежнее задание и `200 OK`.

### `GET /jobs`

История заданий.

```text
GET /jobs?limit=50&offset=0
```

### `GET /jobs/{request_id}`

Полная информация о задании.

### `POST /jobs/{request_id}/cancel`

Отмена активного задания.

### `POST /jobs/{request_id}/retry`

Повторный запуск `failed` или `cancelled` задания.

## 9. Формат имени файла

Текущая реализация формирует имя примерно так:

```text
UPD_20260707_153000_IE_001_47.pdf
```

Общий формат:

```text
TYPE_YYYYMMDD_HHMMSS_USER_NUMBER_CC.pdf
```

Где:

- `TYPE` — код типа документа;
- `YYYYMMDD` — дата;
- `HHMMSS` — время;
- `USER` — код пользователя;
- `NUMBER` — очищенный номер документа;
- `CC` — короткий контрольный код.

Окончательное правило даты для интеграции с Planfix ещё предстоит зафиксировать. Пользователь подтвердил, что дата документа должна браться из поля `Дата составления`; текущая функция архивирования пока использует момент обработки, поэтому это потребуется изменить на следующем этапе.

## 14. Текущие ограничения

### Один worker

Нельзя запускать несколько Uvicorn workers.

```text
--workers 2
```

использовать пока нельзя, потому что очередь и активные задачи находятся в памяти одного процесса.

### Один сканер

Текущая очередь рассчитана на один физический сканер.

### Нет прямого запуска Epson

Текущий backend только ждёт PDF в папке.

Для Epson WorkForce DS-790WN позднее потребуется отдельная реализация `EpsonScannerController` либо управление установленным ПО/драйвером Epson на сервере.

### Нет Planfix-адаптера

Специальный endpoint Planfix пока не реализован.

### Нет обратного обновления Planfix

Имя файла и статус пока не записываются в поля задачи Planfix.

### Нет API-ключа

Авторизация внешних запросов пока не внедрена.

### Нет HTTPS

Локальный адрес `127.0.0.1` не доступен облачному Planfix.

### Нет OCR и AI

Интеллектуальный анализ отложен.

### Нет журнала переходов статусов

Хранится текущее состояние, но не полная история событий.

## 15. Зафиксированный сценарий Planfix

Используется шаблон задачи:

```text
6.1_Внесение документа
```

Поля:

- `Тип документа`;
- `№ документа`;
- `Дата составления`;
- `Сумма документа`;
- `Контрагент`;
- `Имя документа в архиве`.

Типы документов в Planfix:

```text
ДОГ
ЗВЛ
КС2
КС3
НКЛ
СФК
УПД
```

Предлагаемое соответствие:

| Planfix | Архив |
|---|---|
| ДОГ | `DOG` |
| ЗВЛ | `ZVL` |
| КС2 | `KS2` |
| КС3 | `KS3` |
| НКЛ | `NKL` |
| СФК | `SFK` |
| УПД | `UPD` |

Одна задача Planfix соответствует одному документу.

Обычный сценарий:

```text
Пользователь заполняет поля
→ нажимает «Сканировать»
→ Planfix отправляет HTTP-запрос
→ backend создаёт задание
→ запускается или имитируется сканирование
→ PDF переименовывается
→ имя возвращается в Planfix
```

## 16. Ближайший план разработки

### Этап 1. Planfix endpoint

Добавить:

```text
POST /integrations/planfix/scan
```

Он будет:

- принимать данные задачи;
- преобразовывать тип документа;
- создавать внутренний `ScanRequest`;
- возвращать `200 OK`;
- использовать идемпотентность;
- запускать текущую обработку через папку.

### Этап 2. Модели интеграции

Создать:

```text
app/integrations/planfix.py
```

С моделями:

- входящего запроса;
- ответа;
- преобразования типов;
- подготовки внутреннего задания.

### Этап 3. Имитация сканера

Пока без Epson:

```text
Planfix / Swagger
→ backend
→ вручную положить PDF в data/incoming
→ архивирование
```

### Этап 4. Возврат результата в Planfix

Добавить клиент обратного уведомления и запись:

- имени документа;
- статуса;
- ошибки;
- request ID.

### Этап 5. Публичный HTTPS

Настроить тестовый туннель или серверный HTTPS-адрес.

### Этап 6. Кнопка Planfix

Создать кнопку `Сканировать` и HTTP-запрос к backend.

### Этап 7. Epson

На сервере проверить:

- Epson Scan 2;
- сканер по фиксированному IP;
- пустой ADF;
- автоматическую подачу;
- возможность запуска без открытого интерфейса;
- права служебной учётной записи.

### Этап 8. AI

После завершения обычного сканирования:

- OCR;
- определение типа;
- извлечение полей;
- кнопка `Сканировать с ИИ`.
