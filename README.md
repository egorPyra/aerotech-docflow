# Aerotech Docflow

Backend-сервис для запуска и сопровождения процесса сканирования документов.

Сервис принимает HTTP-запрос, создаёт задание, ожидает появление нового PDF-файла в папке сканера, проверяет файл, формирует итоговое имя, вычисляет SHA-256, перемещает документ в архив и сохраняет состояние задания в SQLite.

## Текущий статус

На текущем этапе реализована рабочая backend-цепочка:

```text
POST /scan
→ создание задания
→ постановка в очередь
→ ожидание нового PDF
→ проверка готовности файла
→ проверка PDF-заголовка
→ формирование имени
→ вычисление SHA-256
→ перемещение в архив
→ сохранение результата в SQLite
→ статус done
```

Также реализованы автоматические тесты. На текущем этапе успешно проходят **29 тестов**.

## Назначение модулей

### `app/main.py`

Точка входа FastAPI-приложения.

Содержит HTTP endpoints:

- `GET /`
- `GET /health`
- `POST /scan`
- `GET /jobs`
- `GET /jobs/{request_id}`

Также при запуске сервиса помечает незавершённые задания как `failed`.

### `app/schemas.py`

Содержит Pydantic-модели:

- входящий JSON для `POST /scan`;
- ответ после создания задания;
- полную модель задания;
- допустимые статусы задания.

### `app/repository.py`

Отвечает за работу с SQLite:

- создание таблиц;
- создание заданий;
- чтение задания;
- получение истории;
- обновление статусов;
- сохранение путей к файлам;
- сохранение SHA-256;
- обработку незавершённых заданий после перезапуска.

### `app/service.py`

Содержит основной процесс обработки:

- постановку задания в очередь;
- блокировку одного физического сканера;
- ожидание PDF;
- архивирование;
- обработку ошибок;
- обновление статусов.

### `app/scanner.py`

Наблюдает за папкой сканера.

Модуль:

- запоминает файлы, существовавшие до запуска задания;
- ищет новый PDF;
- проверяет стабильность размера и времени изменения;
- проверяет PDF-заголовок;
- завершает ожидание по таймауту.

Модуль пока не управляет физическим сканером напрямую.

### `app/storage.py`

Отвечает за файловое хранилище:

- очистку частей имени;
- формирование итогового имени;
- вычисление контрольного кода;
- вычисление SHA-256;
- создание структуры архива;
- безопасное перемещение файла;
- защиту от перезаписи.

### `app/config.py`

Содержит настройки приложения:

- папка входящих PDF;
- корень архива;
- путь к SQLite;
- таймаут ожидания;
- интервал проверки папки;
- количество проверок стабильности файла.

## API

### `GET /`

Возвращает информацию о сервисе.

Пример ответа:

```json
{
  "service": "aerotech-docflow",
  "status": "ok",
  "health": "/health",
  "docs": "/docs",
  "jobs": "/jobs"
}
```

### `GET /health`

Проверяет доступность backend-сервиса.

Пример ответа:

```json
{
  "status": "ok"
}
```

### `POST /scan`

Создаёт новое задание и запускает ожидание PDF.

Пример запроса:

```json
{
  "task_id": 52418,
  "document_type": "UPD",
  "document_number": "2455/1",
  "user_code": "IV",
  "context": {
    "source": "planfix"
  }
}
```

Поля:

| Поле | Обязательное | Описание |
|---|---:|---|
| `task_id` | да | Идентификатор задачи во внешней системе |
| `document_type` | да | Код типа документа |
| `document_number` | да | Номер документа |
| `user_code` | да | Код пользователя |
| `context` | нет | Дополнительные данные |

Пример ответа:

```json
{
  "status": "accepted",
  "request_id": "7e17371b-9cc4-431e-9d25-0633786093a6",
  "status_url": "/jobs/7e17371b-9cc4-431e-9d25-0633786093a6"
}
```

HTTP-код:

```text
202 Accepted
```

### `GET /jobs/{request_id}`

Возвращает текущее состояние задания.

Пример ответа:

```json
{
  "request_id": "7e17371b-9cc4-431e-9d25-0633786093a6",
  "task_id": 52418,
  "document_type": "UPD",
  "document_number": "2455/1",
  "user_code": "IV",
  "context": {
    "source": "planfix"
  },
  "status": "done",
  "created_at": "2026-07-06T12:00:00+00:00",
  "updated_at": "2026-07-06T12:01:02+00:00",
  "source_file": "D:\\PROG_PROJECTS\\aerotech-docflow\\data\\incoming\\scan.pdf",
  "result_file": "D:\\PROG_PROJECTS\\aerotech-docflow\\data\\archive\\UPD\\2026\\07\\UPD_20260706_150602_IV_2455-1_32.pdf",
  "result_filename": "UPD_20260706_150602_IV_2455-1_32.pdf",
  "sha256": "64-символьная SHA-256 строка",
  "error": null
}
```

Если задание не найдено:

```json
{
  "detail": "Задание не найдено"
}
```

HTTP-код:

```text
404 Not Found
```

### `GET /jobs`

Возвращает историю заданий.

Поддерживаемые параметры:

| Параметр | По умолчанию | Ограничения |
|---|---:|---|
| `limit` | 50 | от 1 до 200 |
| `offset` | 0 | не меньше 0 |

Пример:

```text
GET /jobs?limit=20&offset=0
```

## Статусы задания

| Статус | Значение |
|---|---|
| `accepted` | Задание создано и ожидает запуска |
| `waiting_for_file` | Сервис ждёт новый PDF |
| `file_received` | PDF найден и принят |
| `archiving` | Идёт вычисление хеша и перенос в архив |
| `done` | Обработка завершена |
| `failed` | Произошла ошибка |

## Как проходит обработка

1. Клиент отправляет `POST /scan`.
2. Сервис создаёт запись в SQLite.
3. Заданию присваивается уникальный `request_id`.
4. Задание встаёт в очередь.
5. После получения блокировки сканера статус меняется на `waiting_for_file`.
6. Сервис запоминает PDF, которые уже находятся в `data/incoming`.
7. Новый PDF должен появиться после создания задания.
8. Сервис ждёт, пока файл перестанет изменяться.
9. Проверяется наличие PDF-заголовка `%PDF-`.
10. Статус меняется на `file_received`.
11. Формируется итоговое имя.
12. Вычисляется SHA-256.
13. Файл перемещается в архив.
14. Результат сохраняется в SQLite.
15. Статус меняется на `done`.

## Папка входящих файлов

По умолчанию сервис ждёт документы здесь:

```text
data/incoming
```

Важно:

- PDF нужно помещать после вызова `POST /scan`;
- уже существующие файлы игнорируются;
- файл должен быть настоящим PDF;
- на один активный запрос должен приходиться один новый PDF.

## Архив

По умолчанию документы сохраняются в:

```text
data/archive
```

Структура архива:

```text
data/archive/
└── UPD/
    └── 2026/
        └── 07/
            └── UPD_20260706_150602_IV_2455-1_32.pdf
```

## Формат имени документа

```text
ККК_ГГГГММДД_ЧЧММСС_П_NNNNN_КС.pdf
```

Пример:

```text
UPD_20260706_150602_IV_2455-1_32.pdf
```

Где:

| Часть | Описание |
|---|---|
| `UPD` | Код типа документа |
| `20260706` | Дата создания архивного файла |
| `150602` | Время создания архивного файла |
| `IV` | Код пользователя |
| `2455-1` | Номер документа после очистки |
| `32` | Короткий контрольный код |

Недопустимые для Windows символы заменяются.

Например:

```text
2455/1 → 2455-1
```

Если имя уже существует, добавляется суффикс:

```text
document.pdf
document_01.pdf
document_02.pdf
```

## SQLite

Файл базы данных:

```text
data/aerotech.db
```

Во время работы также могут появляться:

```text
data/aerotech.db-wal
data/aerotech.db-shm
```

Это служебные файлы SQLite в WAL-режиме.

После перезапуска завершённые задания остаются доступными.

Незавершённые задания автоматически получают:

```text
status = failed
```

и ошибку:

```text
Обработка была прервана перезапуском backend-сервиса
```

Это сделано для того, чтобы новый PDF случайно не был привязан к старому заданию.

## Ограничения текущей версии

### Один процесс Uvicorn

Сейчас блокировка сканера хранится внутри процесса Python.

Не следует запускать приложение так:

```text
uvicorn ... --workers 2
```

Несколько workers не будут разделять одну блокировку и могут обработать один файл несколько раз.

### Один физический сканер

Текущая архитектура предполагает один источник PDF и одну очередь заданий.

### Scan-to-folder

Сервис не нажимает кнопку сканирования и не обращается к TWAIN/WIA.

Предполагается, что сканер или пользователь сохраняет PDF в папку `data/incoming`.

### Нет защиты от повторного POST

Повторный одинаковый запрос пока создаёт новое задание.

### Нет авторизации

Любой клиент, имеющий доступ к endpoint, может создать задание.

## Рекомендации для разработки

- Не хранить рабочие PDF в Git.
- Не добавлять SQLite-базу в репозиторий.
- Не запускать несколько workers.
- Перед изменениями запускать тесты.
- После изменений снова запускать тесты.
- Не привязывать один PDF к нескольким заданиям.
- Не удалять исходный PDF вручную во время обработки.
