# 1. ОБЩИЙ ПРИНЦИП

Система работает как цепочка: `Planfix → запрос → локальный сервис → сканер → файл → сохранение → Planfix`.

Любой сбой **НЕ должен ломать процесс полностью**. Система должна:
- фиксировать состояние;
- уметь продолжать или повторять шаг;
- сообщать человеку, что происходит.

## 2. ОСНОВНЫЕ КЛАССЫ ОШИБОК

| Класс ошибки | Проблема | Причины | Решение / статус |
|---|---|---|---|
| **2.1 Ошибки запуска процесса** | Пользователь нажал «Сканировать», но ничего не произошло. | Локальный сервис не запущен;<br>нет соединения Planfix → сервис;<br>ошибка HTTP запроса. | Вернуть ошибку в Planfix;<br>добавить комментарий в задачу: «Сервис недоступен»;<br>статус: `ERROR_SERVICE_UNAVAILABLE`. |
| **2.2 Ошибки сканера** | Сканирование не началось или зависло. | Сканер отключен;<br>сканер занят;<br>драйвер не отвечает;<br>нет бумаги. | Повторная попытка (retry);<br>таймаут ожидания;<br>статус: `ERROR_SCANNER_NOT_READY`;<br>комментарий в Planfix: «Сканер недоступен или занят». |
| **2.3 Ошибки получения файла** | Скан вроде был, но файл не появился. | Сканер не сохранил PDF;<br>задержка записи файла;<br>ошибка драйвера. | Ожидание файла (polling 5–30 сек);<br>повтор проверки;<br>статус: `WAITING_FOR_FILE`;<br>если не появился → `ERROR_FILE_NOT_CREATED`. |
| **2.4 Ошибки файла (повреждение / дубликаты)** | Файл есть, но он неправильный. | Битый PDF;<br>повторный скан того же документа;<br>конфликт имени файла. | Проверка уникальности имени;<br>генерация уникального ID;<br>статус: `ERROR_DUPLICATE_FILE` или `ERROR_CORRUPTED_FILE`. |
| **2.5 Ошибки хранения** | Файл не сохранился или не синхронизировался. | Нет доступа к папке;<br>Яндекс Диск не синхронизировал;<br>файл заблокирован системой. | Сохранять локально сначала;<br>затем копировать в sync папку;<br>повторная попытка сохранения;<br>статус: `ERROR_STORAGE_FAILED`. |
| **2.6 Ошибки Planfix** | Система не смогла обновить задачу. | API Planfix недоступен;<br>timeout;<br>неверный task_id. | Повтор запроса (retry 2–3 раза);<br>логирование ошибки;<br>статус: `ERROR_PLANFIX_UPDATE_FAILED`. |
| **2.7 Ошибки повторного запуска (дубликаты)** | Пользователь нажал «Сканировать» несколько раз. | Нет блокировки задачи;<br>параллельные запросы. | Блокировка задачи (lock);<br>проверка active scan;<br>игнор повторных запросов;<br>статус: `SCAN_ALREADY_IN_PROGRESS`. |
| **2.8 Человеческие ошибки** | Оператор делает неверные действия. | Не положил документ в сканер;<br>выбрал неправильный тип;<br>нажал кнопку без бумаги. | Явные сообщения в Planfix;<br>понятные статусы: «нет документа», «ожидание бумаги»;<br>не падать, а ждать. |

## 3. ПРИНЦИП ОБРАБОТКИ ОШИБОК

> **Главное правило:** система НЕ падает — она переходит в состояние ошибки.

Вместо `crash` или `exception` без обработки:

```text
SCAN → ERROR_STATE → retry / manual fix
```

## 4. МОДЕЛЬ СОСТОЯНИЙ

**Основной поток**

```text
REQUEST_RECEIVED → SCANNING → WAITING_FOR_FILE → FILE_RECEIVED
→ STORAGE_SAVING → PLANFIX_UPDATING → DONE
```

**Ошибочные состояния**

```text
ERROR_SERVICE_UNAVAILABLE
ERROR_SCANNER_NOT_READY
ERROR_FILE_NOT_CREATED
ERROR_STORAGE_FAILED
ERROR_PLANFIX_UPDATE_FAILED
ERROR_DUPLICATE_REQUEST
```

## 5. ПОВЕДЕНИЕ СИСТЕМЫ ПРИ ОШИБКАХ

Каждая ошибка должна:
1. записываться в лог;
2. отправляться в Planfix как комментарий;
3. менять статус задачи.

Пример:

```text
[ERROR_SCANNER_NOT_READY]
Сканер не отвечает или занят.
Попробуйте позже.
```

## 6. ПРИНЦИП ПОВТОРА (RETRY)

Система должна уметь повторить:
- скан;
- запись файла;
- отправку в Planfix.

Повторы не должны быть бесконечными: максимум **2–3 попытки**.

## 7. УНИКАЛЬНОСТЬ ФАЙЛОВ (КРИТИЧНО)

Каждый файл должен иметь UUID:

```text
uuid.pdf
```

> Цель: никогда не перезаписывать существующие документы.

## 8. КЛЮЧЕВАЯ ИДЕЯ ВСЕЙ СИСТЕМЫ

> Процесс всегда управляемый, даже при сбое.
